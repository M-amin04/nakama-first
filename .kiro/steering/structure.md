# Project Structure

## Root Layout

```
nakama-project-template/
├── src/              # TypeScript source code
├── build/            # Compiled JavaScript output
├── model/            # TensorFlow ML models
├── node_modules/     # NPM dependencies
├── .husky/           # Git hooks
├── .kiro/            # Kiro IDE configuration
└── .git/             # Git repository
```

## Source Directory (`src/`)

Core server-side game logic written in TypeScript:

- **`main.ts`**: Entry point and module initialization
  - Registers RPC functions with Nakama
  - Registers hooks (e.g., after authentication)
  - Initializes match handlers
  - Exposes InitModule function to Nakama runtime

- **`task.ts`**: Shop and inventory system
  - RPC handlers for shop items, inventory, purchases
  - Authentication hooks

- **`match_handler.ts`**: Authoritative match logic (commented out)
  - Match lifecycle functions (init, join, leave, loop, terminate)
  - Turn-based game state management

- **`match_rpc.ts`**: Match-finding RPC functions
- **`daily_rewards.ts`**: Daily reward system
- **`messages.ts`**: Shared type definitions and message schemas
- **`ai.ts`**: AI player integration

## Build Output (`build/`)

Contains the bundled JavaScript file (`index.js`) that Nakama loads at runtime. This is generated by Rollup from the TypeScript source.

## Model Directory (`model/`)

TensorFlow SavedModel format for ML inference via TensorFlow Serving:
- `model/01/`: Versioned model directory
- Contains `.pb` files and variables for model weights

## Configuration Files

### Build & Type System
- `tsconfig.json`: TypeScript compiler options
- `rollup.config.js`: Module bundler configuration
- `babel.config.json`: Babel transpiler settings

### Package Management
- `package.json`: NPM dependencies and scripts
- `package-lock.json`: Locked dependency versions

### Docker
- `Dockerfile`: Nakama server container build instructions
- `docker-compose.yml`: Multi-container orchestration
- `local.yml`: Nakama server configuration

### Code Quality
- `.gitignore`: Git exclusions
- `.husky/`: Git hook scripts for pre-commit and commit-msg

## Coding Conventions

### File Organization
- One module per file
- Entry point must export or register with Nakama initializer
- Commented-out code preserved for reference (match handlers, AI)

### Naming Patterns
- RPC function names: lowercase with underscores (e.g., `get_shop_items`)
- Handler functions: camelCase (e.g., `afterAuthenticateDevice`)
- Constants: camelCase or UPPER_SNAKE_CASE
- Interfaces and types: PascalCase

### Imports
- Use `.js` extensions in import paths (ES module spec compliance)
- Nakama runtime types: `nkruntime.*`

### Function Registration
- RPC functions registered via `initializer.registerRpc(id, handler)`
- Hooks registered via `initializer.registerAfter*` methods
- Match handlers registered via `initializer.registerMatch(name, handlers)`
